## Что это за проект

Это дипломный проект по сегментации клиентов e-commerce и прогнозированию ухудшения клиентского состояния во времени.

В проекте две главные исследовательские линии:

- [client pivot.ipynb](/Users/dl3/Documents/ДИПЛОМ%204%20КУРС%20%28Сегментация%29/client%20pivot.ipynb): построение клиентской витрины и тестирование разных методов кластеризации.
- [segment overflows.ipynb](/Users/dl3/Documents/ДИПЛОМ%204%20КУРС%20%28Сегментация%29/segment%20overflows.ipynb): построение панели `client × quarter`, присвоение RFM-сегментов и прогноз downgrade клиента в следующем квартале.

Коротко:

- первый ноутбук отвечает на вопрос: "какие типы клиентов есть в данных?";
- второй ноутбук отвечает на вопрос: "кто из клиентов перейдет в более слабый сегмент в следующем квартале?".

### Как формируется клиентская витрина

После фильтра по доставленным заказам:

- `Отменено == "Нет"`
- `Статус == "Доставлен"`
- `КоличествоПроданоКлиенту > 0`

строится витрина на уровне клиента (`Телефон_new`) со следующими блоками признаков:

- базовая активность:
  - `delivered_orders`
  - `items`
  - `revenue`
  - `margin`
  - `avg_order_value`
- временные признаки:
  - `first_purchase`
  - `last_purchase`
  - `recency_days`
  - `days_since_first_purchase`
- структура заказа:
  - `most_freq_order_size`
- товарные доли:
  - доли категорий `Группа2` по количеству проданных единиц;
- региональные доли:
  - доли макрорегионов по количеству товаров;
- канальные доли:
  - доли методов доставки по числу заказов;
  - доли способов оплаты по числу заказов;
- качество заказов:
  - `total_orders`
  - `cancel_ratio`
  - `return_ratio`
  - `delivered_ratio`

### Отдельные шаги feature engineering

- delivery methods частично укрупняются:
  - `евросеть/связной -> другое`
  - `dpd -> другое`
  - `транспортная компания -> другое`
- после сборки витрина переупорядочивается в явный `new_order`;
- выбросы удаляются робастно по:

### Какие методы кластеризации пробуются

#### K-means

- оценивается число кластеров через несколько метрик;
- в рабочем варианте используется `k_opt = 17`;
- метки сохраняются в колонку `k_means_10`, хотя это имя не совпадает с фактическим `k=17`;
- строится сводка по кластерам и визуализации через `PCA`.

Практический смысл:

- это главный статичный baseline сегментации в ноутбуке;
- итоговая сводка может экспортироваться в [k-means.xlsx](/Users/dl3/Documents/ДИПЛОМ%204%20КУРС%20%28Сегментация%29/k-means.xlsx).

#### EM / Gaussian Mixture

- подбирается `k` по `BIC`, `AIC`, `log-likelihood`;
- отдельно есть переборы `k` в разных диапазонах;
- в одной из рабочих конфигураций используется `n_clusters = 11`;
- считается soft-assignment и certainty по вероятностям.

Важно:

- в этом блоке есть явный черновой/ошибочный фрагмент:
  - `em_df = client_dfclient_df_positive_margin.copy()`
- значит EM-часть стоит воспринимать как экспериментальную и требующую доочистки перед повторным запуском.

#### DBSCAN

- сначала делается `PCA(n_components=25)`;
- `eps` подбирается через `k-distance graph`;
- рабочий вариант:
  - `eps = 2.5`
  - `min_samples = 70`
- затем строятся:
  - метрики качества по ненулевым кластерам;
  - `t-SNE` и `UMAP` визуализации;
  - агрегированный pivot по кластерам.

Результат экспортируется в:

- [dbscan.xlsx](/Users/dl3/Documents/ДИПЛОМ%204%20КУРС%20%28Сегментация%29/dbscan.xlsx)

#### Hierarchical clustering

- есть exploratory-блок с Ward linkage;
- выглядит незавершённым и не является основным итоговым методом.

#### RFM

- в конце ноутбука есть отдельный блок RFM-сегментации;
- он не является главным артефактом этого ноутбука, но тематически связывает его со вторым ноутбуком.

## RFM-логика во втором ноутбуке

### Базовые правила сегментации

RFM строится на:

- `recency_days`
- `total_orders`
- `margin`

Сначала считаются:

- `R_score`
- `F_score`
- `M_score`
- `RFM_sum`
- `RFM_code`

Затем клиенту присваивается сегмент из набора:

- `Champions`
- `Loyal`
- `Big Spenders`
- `New`
- `Promising`
- `At Risk`
- `Lost`
- иначе `Regular`

### Важная оговорка

Перед расчётом сегментов из данных исключается `2017Q4`, чтобы:

- не искажать квантильные пороги;
- не использовать квартал, для которого нет наблюдаемого следующего квартала в текущем датасете.

## Как устроен переходный анализ сегментов

### Transition panel

После присвоения RFM-сегментов строится панель переходов:

- текущий сегмент `segment`
- следующий сегмент `segment_next`

Логика:

- строки сортируются по `Телефон_new, quarter`;
- `segment_next` считается через `groupby(...).shift(-1)`.

### Train/test split по времени

Текущая рабочая схема во втором ноутбуке такая:

- train:
  - переходы, стартующие в `Q1`
- test:
  - переходы, стартующие в `Q2`

То есть идея такая:

- учимся на `Q1 -> Q2`
- проверяемся на `Q2 -> Q3`

Это справедливо и для Markov baseline, и для ML-подходов.

## Модели во втором ноутбуке

### 1. Markov baseline

Что делает:

- строит матрицу переходов `P(segment_next | segment)` только по текущему сегменту;
- оценивает baseline accuracy и log-loss на тестовом квартале.

Интерпретация в ноутбуке:

- сегменты умеренно нестабильны;
- `At Risk` выглядит центральным "дрейфовым" состоянием;
- `Lost` не является полностью поглощающим состоянием.

### 2. Multiclass prediction of next segment

Цель:

- предсказать `segment_next`, используя:
  - текущий сегмент;
  - квартальные поведенческие признаки;
  - финансовые признаки;
  - структуры категорий, региона, доставки и оплаты.

Текущий набор подходов:

- Multinomial Logistic Regression
- LightGBM multiclass

Признаки:

- базовые квартальные признаки;
- one-hot encoding текущего `segment`.

### 3. Binary classification of downgrade

Это основная постановка для практического риска.

Сначала сегментам задаётся ранговая шкала:

- `Lost = 0`
- `At Risk = 1`
- `Regular = 2`
- `New = 3`
- `Big Spenders = 4`
- `Loyal = 4`
- `Champions = 5`

Затем:

- `downgrade = 1`, если `rank_next < rank_t`
- иначе `downgrade = 0`

То есть downgrade означает переход клиента в более слабое состояние относительно текущего квартала.

### Какие модели сравниваются для downgrade

- Logistic Regression
- LightGBM
- Random Forest
- XGBoost
- SVM

Дополнительно:

- строится список `risk_clients` с вероятностью `p_downgrade`;
- анализируются top-risk клиенты;
- смотрятся feature importances.

## Нюансы и caveats

### 1. Ноутбуки исследовательские, а не production-пайплайн

В обоих ноутбуках есть:

- дублирующие ячейки;
- временные проверки;
- ручные комментарии;
- ячейки с установкой пакетов;
- несколько альтернативных веток экспериментов.

Это нормально для дипломной исследовательской работы, но важно помнить при автоматизации.

### 2. Есть дубли и разные версии артефактов

- `client_pivot_no_id.csv` в корне и в `data/` совпадают;
- `k-means.xlsx` в корне и `data/k-means.xlsx` не совпадают по содержимому;
- часть файлов в корне и в `data/transactions` логически пересекаются, но используются разными ноутбуками.

### 3. Есть несколько мест, которые выглядят черновыми

- в `client pivot.ipynb` имя `k_means_10` не соответствует фактическому `k=17`;
- в EM-блоке есть явный typo:
  - `client_dfclient_df_positive_margin`
- в DBSCAN-блоке есть активные эксперименты с `UMAP`, reinstall пакетов и альтернативные визуализации;
- в первом ноутбуке есть markdown-заметка про аномалии дат, но сама логика обработки таких строк не доведена до отдельного feature/фильтра.

### 4. Второй ноутбук методологически ближе к финальной прикладной задаче

Если в будущем нужно выбирать одно "основное" направление проекта, то оно, скорее всего, здесь:

- [segment overflows.ipynb](/Users/dl3/Documents/ДИПЛОМ%204%20КУРС%20%28Сегментация%29/segment%20overflows.ipynb)

Потому что он уже выходит на:

- прогноз следующего состояния;
- риск downgrade;
- ранжирование клиентов по вероятности ухудшения.

## Как Codex должен использовать этот файл дальше

Если в будущей сессии нужно быстро восстановить контекст проекта, то базовая стратегия такая:

1. Сначала открыть этот файл.
2. Затем, в зависимости от задачи:
   - для кластеризации идти в `client pivot.ipynb` и `data/client_pivot.csv`;
   - для downgrade идти в `segment overflows.ipynb` и `data/client_pivot_quarter.csv`.
3. Считать ноутбуки источником методологии, а `.csv/.parquet` — рабочими артефактами для повторного анализа и моделей.

## Короткая рабочая формулировка проекта

Проект строит клиентские признаки из транзакций интернет-магазина, сначала выделяет интерпретируемые сегменты покупателей, а затем моделирует переходы между сегментами по кварталам и предсказывает риск downgrade клиента в следующем квартале.
