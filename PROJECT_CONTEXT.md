# Project Context

Последнее обновление: 2026-04-06

## Что это за проект

Это дипломный проект по сегментации клиентов e-commerce и прогнозированию ухудшения клиентского состояния во времени.

В проекте две главные исследовательские линии:

- [client pivot.ipynb]: построение клиентской витрины и тестирование разных методов кластеризации.
- [segment overflows.ipynb]: построение панели `client × quarter`, присвоение RFM-сегментов и прогноз downgrade клиента в следующем квартале.

Коротко:

- первый ноутбук отвечает на вопрос: "какие типы клиентов есть в данных?";
- второй ноутбук отвечает на вопрос: "кто из клиентов перейдет в более слабый сегмент в следующем квартале?".

## Логика проекта по уровням

### Уровень 1. Статичная клиентская сегментация

Источник: [client pivot.ipynb]

Цель:

- из транзакций собрать понятную клиентскую витрину;
- попробовать несколько методов кластеризации;
- интерпретировать сегменты по финансовому и поведенческому профилю.

### Уровень 2. Динамика сегментов по кварталам

Источник: [segment overflows.ipynb]

Цель:

- перейти от одной "фотографии" клиента к временному ряду `client × quarter`;
- назначать клиенту RFM-сегмент в каждом квартале;
- моделировать переход `segment -> segment_next`;
- отдельно решать бинарную задачу downgrade.

### Как формируется клиентская витрина

После фильтра по доставленным заказам:

- `Отменено == "Нет"`
- `Статус == "Доставлен"`
- `КоличествоПроданоКлиенту > 0`

строится витрина на уровне клиента (`Телефон_new`) со следующими блоками признаков:

- базовая активность:
  - `delivered_orders`
  - `items`
  - `revenue`
  - `margin`
  - `avg_order_value`
- временные признаки:
  - `first_purchase`
  - `last_purchase`
  - `recency_days`
  - `days_since_first_purchase`
- структура заказа:
  - `most_freq_order_size`
- товарные доли:
  - доли категорий `Группа2` по количеству проданных единиц;
- региональные доли:
  - доли макрорегионов по количеству товаров;
- канальные доли:
  - доли методов доставки по числу заказов;
  - доли способов оплаты по числу заказов;
- качество заказов:
  - `total_orders`
  - `cancel_ratio`
  - `return_ratio`
  - `delivered_ratio`

### Отдельные шаги feature engineering

- delivery methods частично укрупняются:
  - `евросеть/связной -> другое`
  - `dpd -> другое`
  - `транспортная компания -> другое`
- после сборки витрина переупорядочивается в явный `new_order`;
- выбросы удаляются робастно по:
  - `revenue`
  - `margin`
  - `avg_order_value`
- правило выброса:
  - используются 10-й и 90-й процентили;
  - отсечение по `3 * IQR`;
  - клиент удаляется, если аномален минимум в 2 ключевых признаках.

### Подготовка к кластеризации

После корреляционного анализа удаляются коррелирующие признаки, затем витрина сохраняется в [data/client_pivot.csv].

Дальше применяются:

- `log1p` для heavy-tail признаков;
- `RobustScaler` для тяжёлых хвостов;
- `StandardScaler` для ratios и долей.

### Какие методы кластеризации пробуются

#### K-means

- оценивается число кластеров через несколько метрик;
- в рабочем варианте используется `k_opt = 17`;
- метки сохраняются в колонку `k_means_10`, хотя это имя не совпадает с фактическим `k=17`;
- строится сводка по кластерам и визуализации через `PCA`.
- итоговая сводка экспортируется в [k-means.xlsx].

Метрики (k=17):
- Silhouette: **0.1968**
- Davies-Bouldin: **1.4165**
- Calinski-Harabasz: **4249.60**

#### EM / Gaussian Mixture

- подбирается `k` по `BIC`, `AIC`, `log-likelihood`;
- финальный вариант: `n_components = 20`;
- считается soft-assignment и certainty по вероятностям.

Метрики (n=20):
- Silhouette: **0.2072**
- Davies-Bouldin: **1.0257**
- Calinski-Harabasz: **2269.34**
- Шум / выбросы: **7929 точек (9.62%)**

Важно: в этом блоке есть явный черновой/ошибочный фрагмент — `em_df = client_dfclient_df_positive_margin.copy()`. EM-часть стоит воспринимать как экспериментальную и требующую доочистки перед повторным запуском.

#### DBSCAN

- сначала делается `PCA(n_components=25)`;
- `eps` подбирается через `k-distance graph`;
- рабочий вариант: `eps=2.5`, `min_samples=70`;
- строятся метрики качества по ненулевым кластерам, t-SNE и UMAP визуализации, агрегированный pivot.
- результат экспортируется в [dbscan.xlsx].

Метрики:
- Кластеров: **80**
- Шум: **35 185 точек (42.7%)**
- Silhouette: **0.3515**
- Davies-Bouldin: **0.9857**
- Calinski-Harabasz: **4888.82**

#### HDBSCAN (две реализации)

**HDBSCAN #1:**

- вход: `X_pca` (PCA-редукция из блока DBSCAN);
- параметры: `min_cluster_size=100`, `min_samples=20`, `metric='euclidean'`, `cluster_selection_method='eom'`, `prediction_data=True`;
- результирующий датафрейм: `hdbscan_df` (колонки `hdbscan`, `margin/revenue`);
- мягкие вероятности: `certainty` через `all_points_membership_vectors(hdb).max(axis=1)`;
- сводка: `cluster_summary_hdb`, экспортируется в `hdbscan.xlsx`;
- визуализация: UMAP 2D, шум (-1) серым, кластеры по `tab20`.

Метрики:
- Кластеров: **80**
- Шум: **35 185 точек (42.7%)**
- Silhouette: **0.3515**
- Davies-Bouldin: **0.9857**
- Calinski-Harabasz: **4888.82**
- Certainty (медиана): **0.043**, среднее: **0.202** (большинство точек с низкой уверенностью — характерно для шумного PCA-входа)

**HDBSCAN #2 (с грид-сёрчем):**

- перед обучением делается грид-сёрч по `min_cluster_size` ∈ [500, 1000, 2000], `min_samples` ∈ [5, 10, 20], `cluster_selection_method` ∈ ['eom', 'leaf']; результат в `grid_df`;
- вход: `X_umap_cluster` — отдельный UMAP с `n_components=10` (не путать с 2D-UMAP для визуализации);
- параметры: `min_cluster_size=1000`, `min_samples=10`, `metric='euclidean'`, `cluster_selection_method='eom'`, `prediction_data=True`;
- результирующий датафрейм: `hdbscan2_df` (колонки `hdbscan2`, `margin/revenue`);
- мягкие вероятности: `certainty2`;
- сводка: `cluster_summary_hdb2`, экспортируется в `hdbscan2.xlsx`.

Метрики:
- Кластеров: **24**
- Шум: **6 975 точек (8.5%)**
- Silhouette: **0.6196**
- Davies-Bouldin: **0.5987**
- Calinski-Harabasz: **69 726.16**
- Certainty (медиана): **0.529**, среднее: **0.574** — значительно лучше HDBSCAN #1

HDBSCAN #2 является лучшим методом кластеризации в ноутбуке по всем трём метрикам.

В ноутбуке два UMAP: `X_umap` (2D, только для отрисовки) и `X_umap_cluster` (10D, для кластеризации во втором HDBSCAN). Не путать.

## Детально: segment overflows.ipynb

### Главная идея ноутбука

Этот ноутбук переводит анализ из статичной формы в временную:

- единица анализа уже не просто клиент;
- единица анализа становится `клиент × квартал`.

На этой панели:

- строятся квартальные признаки;
- считается RFM-сегмент;
- формируется `segment_next`;
- затем решаются задачи multiclass и binary prediction.

### Сборка базы за 2017

Ноутбук читает файлы из [data/transactions], включая распаковку `.rar`, а затем использует функции:

- `parse_months_year_from_name`
- `load_and_filter`
- `build_dataset`

Итог: формируется [data/transactions/transactions_2017.parquet] — главный консолидированный источник для квартального анализа 2017 года.

### Очистка данных

Повторяются идеи из первого ноутбука:

- исправление числовых типов;
- исправление дат;
- очистка `Телефон_new`;
- удаление строк-доставок;
- удаление неизвестных клиентов;
- геомаппинг города в макрорегион;
- удаление строк без сопоставленного региона;
- фильтр по валидной `Маржа`.

Дополнительно: `quarter = df['Дата'].dt.to_period('Q')` — квартальная логика строится по дате покупки `Дата`, а не по `ДатаДоставки`.

### Как формируется квартальная витрина

После фильтра на доставленные заказы (`Отменено == "Нет"`, `Статус == "Доставлен"`, `КоличествоПроданоКлиенту > 0`) строится витрина `Телефон_new × quarter` с признаками:

- объём и активность: `total_orders`, `delivered_orders`, `items`, `most_freq_order_size`
- финансы: `revenue`, `margin`, `avg_order_value`
- временной контекст: `recency_days`, `days_since_first_purchase`
- поведенческие доли: товарные категории, макрорегионы, методы доставки, формы оплаты
- качество заказов: `cancel_ratio`, `return_ratio`, `delivered_ratio`

### Как считаются recency и tenure по кварталам

- для каждого `client × quarter` берётся конец квартала;
- через `merge_asof` ищется последняя покупка до конца квартала;
- затем считаются `recency_days` и `days_since_first_purchase`.

Признаки не подсматривают в будущее относительно конца квартала.

### Финальная квартальная витрина

После корреляционного анализа удаляются `delivery_магазины` и `payment_наличная`. Витрина сохраняется в [data/client_pivot_quarter.csv] — главный вход для прогнозных задач.

## RFM-логика во втором ноутбуке

RFM строится на `recency_days`, `total_orders`, `margin`. Считаются `R_score`, `F_score`, `M_score`, `RFM_sum`, `RFM_code`. Сегменты: Champions, Loyal, Big Spenders, New, Promising, At Risk, Lost, иначе Regular.

Перед расчётом сегментов из данных исключается `2017Q4` — чтобы не искажать квантильные пороги и не использовать квартал без наблюдаемого следующего.

## Как устроен переходный анализ сегментов

### Transition panel

После присвоения RFM-сегментов строится панель переходов (`segment` → `segment_next`) через `groupby(...).shift(-1)`.

### Train/test split по времени

- train: переходы Q1 → Q2
- test: переходы Q2 → Q3

## Модели во втором ноутбуке

### 1. Markov baseline

Строит матрицу переходов `P(segment_next | segment)` только по текущему сегменту.

Метрики на тесте (Q2→Q3):
- Accuracy: **0.2823**
- Log-loss: **1.6726**

Интерпретация: сегменты умеренно нестабильны; `At Risk` — центральное "дрейфовое" состояние; `Lost` не является полностью поглощающим состоянием.

### 2. Multiclass prediction of next segment

Цель: предсказать `segment_next`, используя текущий сегмент + квартальные поведенческие и финансовые признаки + one-hot `segment`.

**Multinomial Logistic Regression:**
- Accuracy: **0.3381**
- Macro F1: **0.1109**

**LightGBM multiclass:**
- Accuracy: **0.2925**
- Macro F1: **0.0941**
- Log-loss: **1.8315**

Multiclass-задача сложная: все модели слабее простого Markov по accuracy, что говорит о высокой стохастичности переходов.

### 3. Binary classification of downgrade

Ранговая шкала сегментов:
- Lost=0, At Risk=1, Regular=2, New=3, Big Spenders=4, Loyal=4, Champions=5

`downgrade = 1`, если `rank_next < rank_t`.

| Модель | Accuracy | ROC-AUC | F1 | Recall |
|---|---|---|---|---|
| Logistic Regression | 0.7753 | **0.9340** | 0.8461 | — |
| LightGBM | 0.7159 | **0.9586** | 0.7963 | — |
| Random Forest | 0.7412 | **0.9610** | 0.8176 | 0.6934 |
| XGBoost | 0.6969 | **0.9566** | 0.7796 | 0.6409 |
| SVM | 0.690 | 0.922 | 0.774 | 0.635 |
| MLP | 0.751 | 0.934 | 0.826 | 0.709 |

Random Forest даёт наибольший ROC-AUC (0.9610) при очень высокой precision (0.9959), но умеренном recall. LightGBM и RF — лучшие по AUC.

Дополнительно строится список `risk_clients` с вероятностью `p_downgrade`, анализируются top-risk клиенты, смотрятся feature importances.

### 4. Survival analysis (анализ выживаемости)

Формулировка: не "случится ли downgrade?", а "через сколько кварталов клиент впервые деградирует?".

Зависимость: `lifelines` (`pip install lifelines`).

#### Подготовка данных

- `surv_base`: панель `Телефон_new × quarter × segment`;
- `first_dg`: первый квартал downgrade по каждому клиенту;
- `survival`: колонки `Телефон_new`, `first_q`, `last_q`, `event_q`, `event` (0/1), `duration` (кварталы);
- `cox_df`: `survival` + признаки из первого квартала наблюдения.

Объём: 260 960 наблюдений, 37 997 событий (event rate 14.6%).

#### Kaplan-Meier (непараметрическая)

- общая кривая: `KaplanMeierFitter()` на всей `survival`;
- стратифицированная: отдельная кривая для каждого `first_segment` при n ≥ 20.

Ключевые результаты:
- Медианное время до downgrade (общее): **3 квартала**
- Выживаемость по таймлайну: t=1 → 88.6%, t=2 → 68.4%, t=3 → 49.9%

#### Cox Proportional Hazards (полупараметрическая)

- `CoxPHFitter(penalizer=0.1)` на `cox_df`;
- интерпретация: `exp(coef) > 1` — ускоряет downgrade, `< 1` — защищает.

Результаты:
- Concordance Index: **0.75**
- Log-likelihood: -442 978.18

Ключевые ковариаты (все p < 0.005):

| Признак | exp(coef) | Направление |
|---|---|---|
| recency_days | 0.99 | защита (чем дольше не покупал — тем меньше риск перехода) |
| delivered_ratio | 0.80 | защита |
| avg_order_value | 1.00 | нейтральный |
| margin | 1.00 | нейтральный |
| cancel_ratio | 1.14 | риск |
| return_ratio | 1.11 | риск |
| total_orders | 1.07 | риск |
| revenue | 1.00 | нейтральный |

Предсказания: `cox_df["median_cox"]`.

#### AFT Weibull (параметрическая)

- `WeibullAFTFitter(penalizer=0.1)` на том же `cox_df`;
- интерпретация: `coef > 0` — удлиняет время до события, `< 0` — сокращает;
- shape parameter rho: **exp(coef) = 2.21** (распределение с возрастающим хазардом).

Результаты:
- Concordance Index: **0.74**
- Log-likelihood: -91 295.55

Предсказания: `cox_df["median_aft"]`.

Финальный шаг — сравнение: `cox_df[["median_cox", "median_aft"]].describe()`. Обе модели согласуются (concordance ~0.74-0.75).

## Нюансы и caveats

### 1. Ноутбуки исследовательские, а не production-пайплайн

В обоих ноутбуках есть дублирующие ячейки, временные проверки, ручные комментарии, ячейки с установкой пакетов, несколько альтернативных веток экспериментов.

### 2. Есть дубли и разные версии артефактов

- `client_pivot_no_id.csv` в корне и в `data/` совпадают;
- `k-means.xlsx` в корне и `data/k-means.xlsx` не совпадают по содержимому.

### 3. Есть несколько мест, которые выглядят черновыми

- в `client pivot.ipynb` имя `k_means_10` не соответствует фактическому `k=17`;
- в EM-блоке есть явный typo: `client_dfclient_df_positive_margin`;
- в DBSCAN-блоке есть активные эксперименты с UMAP, reinstall пакетов и альтернативные визуализации;
- в ноутбуке два разных UMAP: `X_umap` (2D, для визуализации) и `X_umap_cluster` (10D, для кластеризации в HDBSCAN #2) — не путать.

### 4. Второй ноутбук методологически ближе к финальной прикладной задаче

[segment overflows.ipynb] выходит на прогноз следующего состояния, риск downgrade и ранжирование клиентов по вероятности ухудшения.

## Как использовать этот файл

1. Открыть этот файл для быстрого входа в контекст.
2. Для кластеризации — идти в [client pivot.ipynb] и [data/client_pivot.csv].
3. Для downgrade — идти в [segment overflows.ipynb] и [data/client_pivot_quarter.csv].
4. Считать ноутбуки источником методологии, а `.csv/.parquet` — рабочими артефактами.

## Короткая рабочая формулировка проекта

Проект строит клиентские признаки из транзакций интернет-магазина, сначала выделяет интерпретируемые сегменты покупателей (K-means, DBSCAN, HDBSCAN), а затем моделирует переходы между сегментами по кварталам, предсказывает риск downgrade (бинарная классификация) и время до первого downgrade (анализ выживаемости).
